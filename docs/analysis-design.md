# เอกสารข้อกำหนดระบบ (System Requirement Specification)

**โครงการ:** [PrimePC] – ระบบเว็บไซต์จำหน่ายอุปกรณ์คอมพิวเตอร์

| เวอร์ชัน | วันที่แก้ไข | ผู้จัดทำ | รายละเอียดการแก้ไข |
|---|---|---|---|
| 1.0 | [6-7-2026] | [ธเนศวร ศรีทับทิม] | เอกสารฉบับแรก |

---

## 1. ภาพรวมโครงการ (Project Overview)

[PrimePC] คือระบบเว็บไซต์อีคอมเมิร์ซสำหรับจำหน่ายอุปกรณ์คอมพิวเตอร์ (CPU, Mainboard,
การ์ดจอ, RAM, Storage, PSU, เคส ฯลฯ) รองรับทั้งฝั่งลูกค้า (Customer Frontend) และฝั่งผู้ดูแลระบบ
(Admin Dashboard) โดยมีเป้าหมายให้ผู้ใช้สามารถเลือกซื้อสินค้า ตรวจสอบความเข้ากันได้ของอุปกรณ์
ชำระเงิน และติดตามสถานะคำสั่งซื้อได้อย่างสะดวก

## 2. ขอบเขตของระบบ (Scope)

- ระบบรองรับผู้ใช้งาน 2 กลุ่มหลัก: **ลูกค้า (Customer)** และ **ผู้ดูแลระบบ (Admin)**
- ระบบไม่รวมการจัดการคลังสินค้าจริงในโกดัง (Physical Warehouse Management) เชิงลึก
- ระบบรองรับการชำระเงินผ่าน Payment Gateway ที่เชื่อมต่อกับผู้ให้บริการภายนอก

## 3. ความต้องการด้านฟังก์ชันการทำงาน (Functional Requirements)

### 3.1 ฝั่งลูกค้า (Customer Frontend)

| รหัส | รายการ | รายละเอียด | ความสำคัญ |
|---|---|---|---|
| C-01 | สมัครสมาชิก/เข้าสู่ระบบ | ลงทะเบียนบัญชีผู้ใช้ ยืนยันอีเมล เข้าสู่ระบบ | High |
| C-02 | ค้นหาและกรองสินค้า | ค้นหาตามชื่อ/สเปก/หมวดหมู่ ราคา ยี่ห้อ | High |
| C-03 | ดูรายละเอียดสินค้า | แสดงสเปก รูปภาพ รีวิว สต๊อกคงเหลือ | High |
| C-04 | ตัวช่วยเช็คสเปคเข้ากันได้ | ตรวจสอบ Socket CPU/Mainboard, กำลังไฟ PSU ที่ต้องใช้ | Medium |
| C-05 | ตะกร้าสินค้า | เพิ่ม/ลบ/แก้ไขจำนวนสินค้าในตะกร้า | High |
| C-06 | ชำระเงิน | ชำระผ่านบัตร/โอนเงิน/PromptPay | High |
| C-07 | ติดตามคำสั่งซื้อ | ดูสถานะคำสั่งซื้อและการจัดส่ง | High |
| C-08 | ระบบสมาชิก/คะแนนสะสม | สะสมแต้ม/ประวัติการสั่งซื้อ | Medium |
| C-09 | แจ้งเตือน | อีเมล/LINE Notify แจ้งสถานะคำสั่งซื้อ | Medium |

### 3.2 ฝั่งผู้ดูแลระบบ (Admin Dashboard)

| รหัส | รายการ | รายละเอียด | ความสำคัญ |
|---|---|---|---|
| A-01 | จัดการสินค้า | เพิ่ม/แก้ไข/ลบสินค้า จัดการสต๊อก | High |
| A-02 | จัดการคำสั่งซื้อ | อัปเดตสถานะคำสั่งซื้อ/การจัดส่ง | High |
| A-03 | จัดการสมาชิก | ดู/ระงับ/แก้ไขข้อมูลผู้ใช้งาน | Medium |
| A-04 | รายงานยอดขาย | สรุปยอดขาย/สินค้าขายดีรายเดือน | Medium |

## 4. ความต้องการด้านไม่ใช่ฟังก์ชันการทำงาน (Non-Functional Requirements)

- **Performance:** หน้าเว็บโหลดไม่เกิน 3 วินาทีในสภาวะการใช้งานปกติ
- **Security:** เข้ารหัสรหัสผ่าน (Hashing), ใช้ JWT/OAuth2 สำหรับ Authentication
- **Scalability:** รองรับผู้ใช้งานพร้อมกันจำนวนมากในช่วงโปรโมชัน (เช่น 11.11, Black Friday)
- **Usability:** รองรับ Responsive Design ใช้งานได้ทั้งบนเดสก์ท็อปและมือถือ

## 5. การวิเคราะห์สถาปัตยกรรมตามองค์ประกอบหลัก (Software Architecture)

### 5.1 Frontend Architecture (ส่วนที่ผู้ใช้งานโต้ตอบกับระบบ)

- แสดงหน้าเว็บไซต์ / รายการสินค้า / รายละเอียดสินค้า
- ระบบสมัครสมาชิก / เข้าสู่ระบบ
- ตะกร้าสินค้าและหน้าชำระเงิน
- Dashboard ผู้ดูแลระบบ

**โครงสร้าง:** Component-Based Architecture / Single Page Application (SPA)
**เทคโนโลยีที่เหมาะสม:** React.js / Next.js, Tailwind CSS
**สิ่งที่ควรพิจารณา:** UX/UI (Responsive, Mobile First), Security (JWT/OAuth2), Scalability (CDN, Caching)

### 5.2 Backend Architecture (ส่วนประมวลผลหลักของระบบ)

- จัดการ Business Logic ของสินค้า/สมาชิก/คำสั่งซื้อ/การชำระเงิน
- เชื่อมต่อ API ภายนอก (Payment Gateway, ระบบขนส่ง)

**โครงสร้าง:** ระยะเริ่มต้นใช้ Monolithic Architecture (ทีมเล็ก พัฒนาเร็ว) เมื่อระบบขยายตัวจึงแยกเป็น
Microservices เช่น Auth Service / Product Service / Order Service / Payment Service

**เทคโนโลยีที่เหมาะสม:** Node.js + Express.js, REST API, API Gateway, Docker, CI/CD (GitHub Actions)

### 5.3 Database Architecture (ระบบจัดเก็บข้อมูล)

- เก็บข้อมูลผู้ใช้ สินค้า คำสั่งซื้อ รายการชำระเงิน ประวัติการสั่งซื้อ

**โครงสร้าง:** Relational Database (MySQL/PostgreSQL) สำหรับข้อมูลธุรกรรมที่ต้องความถูกต้องสูง
ส่วน NoSQL (Redis) ใช้สำหรับ Cache/Session และ Elasticsearch สำหรับการค้นหาสินค้า

## 6. หลักการออกแบบสถาปัตยกรรมที่นำมาประยุกต์ใช้ (Design Principles)

| หลักการ | การนำไปใช้ในโครงงาน |
|---|---|
| Separation of Concerns | แยก Frontend / Backend / Database ออกจากกันชัดเจน |
| Single Responsibility | แยก Service เช่น User_Service, Product_Service, Order_Service |
| Loose Coupling | เชื่อมต่อกันผ่าน REST API แทนการผูกติดกันโดยตรง |
| Scalability | ใช้ Load Balancer และ Horizontal Scaling รองรับผู้ใช้จำนวนมาก |
| Security | เข้ารหัสข้อมูลสำคัญ, Authentication, Secure by Design |
